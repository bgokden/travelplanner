travelplanner — how it works

Every itinerary below is the actual value returned by plan(...), generated from the runnable examples/scenarios.py.

What it does

You give it two places and a departure time. It plans the whole journey — getting to a nearby station or airport, the trains/ferries/flights in between (with real schedules and seasonal availability), and the trip from the final stop to your door — and ranks the options by the objective you choose: fastest, cheapest (CHEAPEST), fewest changes (FEWEST_TRANSFERS), or air-first (AIR_PRIORITY).

Two places + time→ Get to nearby stops→ Scan real schedules→ Get to the door→ Rank by objective→ Itineraries
It only ever uses scheduled connections that actually run on that date and season. If there is no way through, it returns an empty list rather than inventing a sea or air crossing. (The short door-to-stop hops are estimated by road distance.)

How to read the results

You call plan(origin, dest, departure, timetable, connector, objective=...) and get back a ranked list of itineraries. Each itinerary is a sequence of legs (one row below = one leg of the trip):

  • main mode — the dominant vehicle of the trip (the longest leg).
  • arrive / total — door-to-door arrival time and elapsed time.
  • cost band — a relative low / medium / high indicator, not a currency price.
  • changes — how many times you switch vehicles (transfers).
  • +wait — idle time at a stop before that leg departs (e.g. time at the airport before the flight).

An itinerary object exposes legs, depart_at, arrive_at, total_duration, primary_mode, and cost_level; each leg has mode, from_loc, to_loc, travel_time, and overhead (the wait).

Example 1 — A business trip, time matters

"I'm in Zurich and need to be in Berlin. Get me there fast."

results = plan(origin, dest, depart, timetable, connector,
              objective=Objective.AIR_PRIORITY)

The engine found three real ways to make the trip:

flight11:384h 09mhigh0

Example 2 — Same trip, but on a budget

"Same Zurich to Berlin, but I'd rather save money than time."

results = plan(origin, dest, depart, timetable, connector,
              objective=Objective.CHEAPEST)

Same three options, re-ranked — now the train wins:

TRAIN · arrive 16:03 · total 8h 33m · cost medium · 1 change(s)
walkHotel Storchen, Zurich → Zurich HB07:30–07:39
trainZurich HB → Halle (Saale) Hbf08:00–14:30 +20m wait
trainHalle (Saale) Hbf → Berlin Hbf14:50–16:00 +20m wait
carBerlin Hbf → Office in Berlin-Mitte16:00–16:03
Nothing about the trip changed except the objective. The engine keeps the whole trade-off set and just picks differently.

Example 3 — An island, and a ferry that only runs in summer

"We rented a villa on Mljet, off Dubrovnik. There's no bridge — the only way across is the catamaran, and it stops for winter."

Booking for July:

results = plan(origin, dest, datetime(2026, 7, 20, 14, 0), timetable, connector)
FERRY · arrive 17:15 · total 3h 15m · cost medium · 0 change(s)
carOld Town apartment, Dubrovnik → Dubrovnik Port14:00–14:10
ferryDubrovnik Port → Mljet (Sobra)16:00–17:00 +1h 50m wait
carMljet (Sobra) → Villa on Mljet17:00–17:15

The same trip in January:

results = plan(origin, dest, datetime(2026, 1, 20, 14, 0), timetable, connector)
# results == []
No itinerary. The ferry doesn't run in January, and the island has no road across the sea, so the engine returns an empty list rather than a made-up "drive across the water".

Run it yourself

git clone https://github.com/bgokden/travelplanner.git && cd travelplanner
uv sync                                  # install (not yet on PyPI)
uv run travelplanner demo                # the bundled sample, every objective
uv run python examples/scenarios.py      # the three examples above
About these examples. The timetables here are small and illustrative (the engine routes over whatever schedule/road data you give it — e.g. a GTFS feed). Times are naive local; every stop above is within one timezone, and cross-timezone international flights are a known limitation. The cost band is qualitative, not real fares.
